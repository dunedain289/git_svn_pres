Subversion

"CVS done right"

SVN Data Model

  • Sequential, numbered file deltas
  • Database backed for data integrity
  • Version encompases entire repo (not just 1 file)
  • Version numbers assigned by central repo

SVN Basics

Initial working copy setup

$ svn checkout <url>

Getting latest changes

$ svn update

U    mobdispatcher/pli.tab
A    mobdispatcher/cJSON.c
U    mobdispatcher/mtf_dispatcher.c
U    mobdispatcher/Makefile
U    archcov/archcov.cc
U    archcov/archcov.h
U    debug/debug.m4
U    debug/debug.c
D    spd/chiron.make

Checking edit status

$ svn status

?      cscope.out
?      tmp
M      Makefile
D      fpudispatcher/acc_readmem.c
A      fpudispatcher/newfile.c

Committing

$ svn commit

Will open your $EDITOR for a message.

Branching

$ svn copy <trunk URL> <branch URL>

SVN branches are just copies

Merging

Tracking trunk:

$ svn merge <trunk URL>

Integrating into trunk:

$ svn merge --reintegrate <branch URL>

Don't use branch after reintegration

Handling conflicts

$ svn resolved <filename>

When more than 1 person edits the same section of a file, SVN doesn't know how to merge.

It leaves .mine, .theirs, .r389, files for use in resolving.

Also leaves markers in file: <<<<< r1.

Git

Fast & Distributed Version Control

Data Model

  • DAG of directory snapshots
  • Stored completely in .git directory
  • Version encompases entire repo
  • No central server/repo

Graph

Git Basics

Initial working copy setup

$ git init .

$ git clone <url>

Getting latest changes

Simple:

$ git pull

Details:

$ git fetch origin
$ git merge origin/master

Checking edit status

$ git status

Committing

Will open your $EDITOR for a message.

$ git add <files>
$ git commit

OR

$ git commit -a

Branching

$ git checkout -b <new branchname>

OR

$ git branch <new branchname>
$ git checkout <new branchname>

Git branches are just pointers (super cheap!!!)

Merging

$ git merge <other branch>

$ git rebase <other branch>

Handling conflicts

$ git add <fixed file>
$ git commit

When more than 1 person edits the same section of a file, Git doesn't know how to merge.

It leaves .mine, .theirs, .base files for use in resolving.

Also leaves markers in file: <<<<<, ======, >>>>>>.

Sharing

$ git push
$ git push <remote> <local branch>:<remote branch>

Merge vs. Rebase

Similar result, different history in repo.

Merge creates merge commits with 2 parents.

Rebase replays one branch's commits on top of another. Results in a linear history, even if it didn't actually happen that way.

Merge

Rebase

WARNING

REBASE CHANGES HISTORY

Don't do this if you've pushed (shared) the commits you're rebasing.

If you do, you'll cause lots of pain for others when they pull!

Reverting 1 file

$ git checkout -- <filename>

Pulls a file fresh from the last commit.

Reverting everything

$ git reset --hard HEAD

Blows away ALL CHANGES since your last commit.

Also unstages everything.

Undoing a commit with a new commit

$ git revert <bad commit>

This creates a new commit that undoes the changes in the bad commit - always safe.

Throwing away commits

$ git reset --hard <commit-ish to go back to>

This doesn't actually remove the commits from the db, but they're hard to get to after this.

Trying a merge again

$ git reset --hard HEAD^

Resets your branch to it's previous commit (before the merge), so you can try again or wait for the merge.

Major Differences

  • SVN: server required

    Git: commits are local

  • SVN: merge (update) before commit

    Git: merge is separate step

  • SVN: commit without staging

    Git: staging normal, can skip

Major Differences

  • SVN: rough branch tracking

    Git: branch tracking from DAG

  • SVN: commit == sharing

    Git: commit != sharing

  • SVN: central server required

    Git: every working copy is repo

Subversion wins

  • Simpler mental model
  • Better Windows clients
  • Commit doesn't require stage step

Git Wins

  • Excellent branch/merge handling

  • Don't have to merge to commit

  • Don't need central server

  • GitHub
